# Сценарій використання: Мульти-агентний проєкт

## Коли використовувати цей шлях

Використовуйте цей варіант, коли функціональність охоплює кілька доменів (наприклад, backend + frontend + QA) і паралельне виконання є вигідним.

## Модель координації

Рекомендована послідовність:

1. `/plan` для декомпозиції та відображення залежностей
2. `/coordinate` для порядку виконання та призначення відповідальних
3. паралельний `agent:spawn` для кожного домену
4. `/review` для контрольної точки QA/безпеки/продуктивності

## Стратегія сесій та робочих просторів

Використовуйте один ідентифікатор сесії для кожного потоку функціональності:

```text
session-auth-v2
```

Призначайте ізольовані робочі простори для кожного домену, щоб зменшити конфлікти злиття:

- backend: `./apps/api`
- frontend: `./apps/web`
- mobile: `./apps/mobile`

## Приклад запуску

```bash
oh-my-ag agent:spawn backend "Implement JWT auth API + refresh flow" session-auth-v2 -w ./apps/api
oh-my-ag agent:spawn frontend "Build login + refresh UX with error states" session-auth-v2 -w ./apps/web
oh-my-ag agent:spawn qa "Review auth risks, test matrix, and regression scope" session-auth-v2
```

## Правило "контракт насамперед"

Перед паралельним кодуванням зафіксуйте спільні контракти:

- схеми запитів/відповідей
- коди помилок та повідомлення
- припущення щодо життєвого циклу авторизації/сесії

Якщо контракти змінюються під час виконання, призупиніть залежних агентів та повторно надішліть промпти з оновленим контрактом.

## Контрольні точки злиття

Не зливайте, поки всі перевірки не пройдені:

1. доменні тести проходять
2. точки інтеграції відповідають узгодженим контрактам
3. критичні/серйозні проблеми QA вирішені або явно прийняті як ризик
4. changelog або нотатки до релізу оновлені, коли змінюється зовнішньо видима поведінка

## Операційні антипатерни

Уникайте:

- спільного використання одного робочого простору для всіх агентів
- зміни контрактів без повідомлення інших агентів
- злиття backend/frontend окремо до перевірки сумісності

## Критерії завершення

Мульти-агентне виконання завершено, коли:

- заплановані задачі виконані у всіх доменах
- кросдоменна інтеграція валідована
- підтвердження QA (або задокументоване прийняття ризику) зафіксовано

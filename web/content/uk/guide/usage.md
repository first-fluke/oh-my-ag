# Як використовувати навички мультиагентної системи Antigravity

## Швидкий старт

1. **Відкрити в Antigravity IDE**
   ```bash
   antigravity open /path/to/oh-my-ag
   ```

2. **Навички виявляються автоматично.** Antigravity сканує `.agent/skills/` та індексує всі доступні навички.

3. **Спілкуйтесь в IDE.** Опишіть, що ви хочете створити.

---

## Приклади використання

### Приклад 1: Просте одно-доменне завдання

**Ви вводите:**
```
"Створи компонент форми входу з полями для email та паролю, використовуючи Tailwind CSS"
```

**Що відбувається:**
- Antigravity визначає, що це відповідає `frontend-agent`
- Навичка завантажується автоматично (Progressive Disclosure)
- Ви отримуєте React компонент з TypeScript, Tailwind, валідацією форми

### Приклад 2: Складний мультидоменний проєкт

**Ви вводите:**
```
"Побудуй TODO додаток з автентифікацією користувача"
```

**Що відбувається:**

1. **Workflow Guide активується** — виявляє мультидоменну складність
2. **PM Agent планує** — створює розбивку завдань з пріоритетами
3. **Ви створюєте агентів через CLI**:
   ```bash
   oh-my-ag agent:spawn backend "JWT автентифікація API" session-01 &
   oh-my-ag agent:spawn frontend "UI входу та TODO" session-01 &
   wait
   ```
4. **Агенти працюють паралельно** — зберігають результати в Knowledge Base
5. **Ви координуєте** — переглядаєте `.agent/brain/` для узгодженості
6. **QA Agent перевіряє** — аудит безпеки/продуктивності
7. **Виправлення та ітерація** — повторно створюєте агентів з корекціями

### Приклад 3: Виправлення помилок

**Ви вводите:**
```
"Є помилка — при натисканні на вхід з'являється 'Cannot read property map of undefined'"
```

**Що відбувається:**

1. **debug-agent активується** — аналізує помилку
2. **Знайдено коренну причину** — компонент виконує map над `todos` до завантаження даних
3. **Надано виправлення** — додані стани завантаження та перевірки на null
4. **Написано регресійний тест** — забезпечує, що помилка не повернеться
5. **Знайдені схожі патерни** — проактивно виправлені ще 3 компоненти

### Приклад 4: CLI-паралельне виконання

```bash
# Один агент (робочий простір автовизначається)
oh-my-ag agent:spawn backend "Реалізувати JWT auth API" session-01

# Паралельні агенти
oh-my-ag agent:spawn backend "Реалізувати auth API" session-01 &
oh-my-ag agent:spawn frontend "Створити форму входу" session-01 &
oh-my-ag agent:spawn mobile "Побудувати екрани автентифікації" session-01 &
wait
```

**Моніторинг в реальному часі:**
```bash
# Термінал (окреме вікно терміналу)
bunx oh-my-ag dashboard

# Або браузер
bunx oh-my-ag dashboard:web
# → http://localhost:9847
```

## Дашборди в реальному часі

### Термінальний дашборд

```bash
bunx oh-my-ag dashboard
```

Відстежує `.serena/memories/` за допомогою `fswatch` (macOS) або `inotifywait` (Linux). Відображає живу таблицю зі статусом сесії, станами агентів, ходами та останньою активністю. Оновлюється автоматично при зміні файлів пам'яті.

**Вимоги:**
- macOS: `brew install fswatch`
- Linux: `apt install inotify-tools`

### Веб-дашборд

```bash
npm install          # тільки перший раз
bunx oh-my-ag dashboard:web
```

Відкрийте `http://localhost:9847` у вашому браузері. Функції:

- **Оновлення в реальному часі** через WebSocket (подієво-орієнтоване, не опитування)
- **Авто-перепідключення** якщо з'єднання обривається
- **Інтерфейс у стилі Serena** з фіолетовими акцентами
- **Статус сесії** — ID та стан виконання/завершено/невдало
- **Таблиця агентів** — назва, статус (з кольоровими крапками), кількість ходів, опис завдання
- **Лог активності** — останні зміни з файлів прогресу та результатів

Сервер відстежує `.serena/memories/` за допомогою chokidar з дебаунсингом (100ms). Лише змінені файли викликають читання — без повного пересканування.

## Ключові концепції

### Progressive Disclosure (Прогресивне розкриття)
Antigravity автоматично зіставляє запити з навичками. Ви ніколи не вибираєте навичку вручну. Лише потрібна навичка завантажується в контекст.

### Токен-оптимізована архітектура навичок
Кожна навичка використовує двошарову архітектуру для максимальної токен-ефективності:
- **SKILL.md** (~40 рядків): Ідентичність, маршрутизація, основні правила — завантажується негайно
- **resources/**: Протоколи виконання, приклади, чеклісти, інструкції з обробки помилок — завантажується на вимогу

Спільні ресурси знаходяться в `_shared/` (не є навичкою) та на них посилаються всі агенти:
- Протоколи виконання ланцюга міркувань з 4-кроковим робочим процесом
- Few-shot приклади введення/виведення для керівництва моделями середнього рівня
- Інструкції з відновлення після помилок з ескалацією "3 strikes"
- Шаблони міркувань для структурованого багатокрокового аналізу
- Управління бюджетом контексту для рівнів моделей Flash/Pro
- Автоматична перевірка через `verify.sh`
- Накопичення навчання між сесіями

### Створення агентів через CLI
Використовуйте `oh-my-ag agent:spawn` для запуску агентів через CLI. Враховує `agent_cli_mapping` в `user-preferences.yaml` для вибору відповідного CLI (gemini, claude, codex, qwen) для кожного типу агента. Робочий простір автовизначається з типових конвенцій монорепозиторія або може бути встановлений явно з `-w`.

### Knowledge Base
Результати агентів зберігаються в `.agent/brain/`. Містить плани, код, звіти та примітки координації.

### Serena Memory
Структурований стан виконання в `.serena/memories/`. Orchestrator записує інформацію про сесію, дошки завдань, прогрес кожного агента та результати. Дашборди відстежують ці файли для моніторингу.

### Робочі простори
Агенти можуть працювати в окремих директоріях, щоб уникнути конфліктів. Робочий простір автовизначається з типових конвенцій монорепозиторія:
```
./apps/api   або ./backend   → Робочий простір Backend Agent
./apps/web   або ./frontend  → Робочий простір Frontend Agent
./apps/mobile або ./mobile   → Робочий простір Mobile Agent
```

## Доступні навички

| Навичка | Автоактивується для | Вивід |
|---------|-------------------|-------|
| workflow-guide | Складних мультидоменних проєктів | Покрокова координація агентів |
| pm-agent | "сплануй це", "розбий" | `.agent/plan.json` |
| frontend-agent | UI, компоненти, стилізація | React компоненти, тести |
| backend-agent | API, бази даних, авторизація | API ендпоінти, моделі, тести |
| mobile-agent | Мобільні додатки, iOS/Android | Flutter екрани, управління станом |
| qa-agent | "перевір безпеку", "аудит" | Звіт QA з пріоритизованими виправленнями |
| debug-agent | Звіти про помилки, повідомлення про помилки | Виправлений код, регресійні тести |
| orchestrator | Виконання CLI підагентів | Результати в `.agent/results/` |
| commit | "коміт", "зберегти" | Git коміти (авто-розбиття по функціям) |

## Команди робочих процесів

Введіть їх у чаті Antigravity IDE для запуску покрокових робочих процесів:

| Команда | Опис |
|---------|------|
| `/coordinate` | Мультиагентна оркестрація через CLI з покроковим керівництвом |
| `/orchestrate` | Автоматичне CLI-паралельне виконання агентів |
| `/plan` | PM декомпозиція завдань з API контрактами |
| `/review` | Повний QA конвеєр (безпека, продуктивність, доступність, якість коду) |
| `/debug` | Структуроване виправлення помилок (відтворити → діагностувати → виправити → регресійний тест) |

Це окремо від **навичок** (які автоактивуються). Робочі процеси дають вам явний контроль над багатокроковими процесами.

---

## Типові робочі процеси

### Робочий процес A: Одна навичка

```
Ви: "Створи компонент кнопки"
  → Antigravity завантажує frontend-agent
  → Отримуєте компонент негайно
```

### Робочий процес B: Мультиагентний проєкт (Авто)

```
Ви: "Побудуй TODO додаток з автентифікацією"
  → workflow-guide активується автоматично
  → PM Agent створює план
  → Ви створюєте агентів через CLI (oh-my-ag agent:spawn)
  → Агенти працюють паралельно
  → QA Agent перевіряє
  → Виправлення проблем, ітерація
```

### Робочий процес B-2: Мультиагентний проєкт (Явний)

```
Ви: /coordinate
  → Покроковий керований робочий процес
  → Планування PM → перевірка плану → створення агентів → моніторинг → перевірка QA
```

### Робочий процес C: Виправлення помилок

```
Ви: "Кнопка входу викидає TypeError"
  → debug-agent активується
  → Аналіз коренної причини
  → Виправлення + регресійний тест
  → Перевірені схожі патерни
```

### Робочий процес D: CLI оркестрація з дашбордом

```
Термінал 1: bunx oh-my-ag dashboard:web
Термінал 2: oh-my-ag agent:spawn backend "завдання" session-01 &
            oh-my-ag agent:spawn frontend "завдання" session-01 &
Браузер:    http://localhost:9847 → статус в реальному часі
```

## Поради

1. **Будьте конкретними** — "Побудуй TODO додаток з JWT авторизацією, React frontend, FastAPI backend" краще ніж "зроби додаток"
2. **Використовуйте CLI створення** для мультидоменних проєктів — не намагайтесь зробити все в одному чаті
3. **Переглядайте Knowledge Base** — перевіряйте `.agent/brain/` для узгодженості API
4. **Ітеруйте з повторним створенням** — уточнюйте інструкції, не починайте спочатку
5. **Використовуйте дашборди** — `bunx oh-my-ag dashboard` або `bunx oh-my-ag dashboard:web` для моніторингу сесій orchestrator
6. **Окремі робочі простори** — призначте кожному агенту власну директорію

---

## Усунення несправностей

| Проблема | Рішення |
|----------|----------|
| Навички не завантажуються | `antigravity open .`, перевірте `.agent/skills/`, перезапустіть IDE |
| CLI не знайдено | Перевірте `which gemini` / `which claude`, встановіть відсутні CLI |
| Несумісні результати агентів | Перегляньте обидва в Knowledge Base, повторно створіть з корекціями |
| Дашборд: "No agents" | Файли пам'яті ще не створені, спочатку запустіть orchestrator |
| Веб-дашборд не запускається | Запустіть `npm install` для встановлення chokidar та ws |
| fswatch не знайдено | macOS: `brew install fswatch`, Linux: `apt install inotify-tools` |
| QA звіт має 50+ проблем | Спочатку зосередьтесь на CRITICAL/HIGH, решту задокументуйте для потім |

## CLI команди

```bash
bunx oh-my-ag                # Інтерактивний інсталятор навичок
bunx oh-my-ag doctor         # Перевірка налаштування та відновлення відсутніх навичок
bunx oh-my-ag doctor --json  # JSON вивід для CI/CD
bunx oh-my-ag update         # Оновлення навичок до останньої версії
bunx oh-my-ag stats          # Переглянути метрики продуктивності
bunx oh-my-ag stats --reset  # Скинути метрики
bunx oh-my-ag retro          # Ретроспектива сесії (навчання та наступні кроки)
bunx oh-my-ag dashboard      # Термінальний дашборд в реальному часі
bunx oh-my-ag dashboard:web  # Веб-дашборд (http://localhost:9847)
bunx oh-my-ag help           # Показати довідку
```

# Сценарій використання: Виправлення багів

## Формат прийому

Почніть із відтворюваного звіту:

```text
Symptom:
Environment:
Steps to reproduce:
Expected vs actual:
Logs/trace:
Regression window (if known):
```

## Тріаж за серйозністю

Класифікуйте якомога раніше для вибору швидкості реагування:

- `P0`: втрата даних, обхід авторизації, збій продакшену
- `P1`: зламаний основний сценарій користувача
- `P2`: погіршена поведінка з обхідним шляхом
- `P3`: незначний/неблокуючий

`P0/P1` завжди повинні включати рецензування QA/безпеки.

## Цикл виконання

1. Точно відтворити в мінімальному середовищі.
2. Ізолювати першопричину (а не лише усувати симптоми).
3. Реалізувати найменше безпечне виправлення.
4. Додати регресійні тести для проблемного шляху.
5. Повторно перевірити суміжні шляхи, що ймовірно мають той самий тип збою.

## Шаблон промпту для debug-agent

```text
Bug: <error/symptom>
Repro steps: <steps>
Scope: <files/modules>
Expected behavior: <expected>
Need:
1) root cause
2) minimal fix
3) regression tests
4) adjacent-risk scan
```

## Типові сигнали ескалації

Ескалюйте до QA або безпеки, коли баг стосується:

- авторизації/сесії/оновлення токенів
- меж дозволів
- узгодженості платежів/транзакцій
- регресій продуктивності під навантаженням

## Валідація після виправлення

- оригінальне відтворення більше не завершується збоєм
- немає нових помилок у пов'язаних потоках
- тести завершуються збоєм до виправлення та проходять після виправлення
- шлях відкату зрозумілий, якщо потрібен хотфікс

## Критерії завершення

Виправлення багів завершено, коли:

- першопричину ідентифіковано та задокументовано
- виправлення верифіковано через відтворювані перевірки
- регресійне покриття забезпечено
